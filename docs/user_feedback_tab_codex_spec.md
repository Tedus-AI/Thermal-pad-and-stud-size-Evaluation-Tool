# 新增分頁開發規格：使用者建議與改善

## 1. 開發目標

在現有 `Thermal & Component Spec Management Tool` 中新增一個分頁：

**使用者建議與改善**

此分頁的目標是提供跨部門使用者在操作工具時，可以記錄：

- Bug 回報
- 改善建議
- 新功能需求
- 資料欄位問題
- 權限 / 登入 / 儲存問題

並讓工具維護者可以追蹤每一筆回饋的處理狀態。

此功能不是單純留言板，而是簡化版的 **Issue / Feedback Tracker**。

---

## 2. 新增分頁位置

請在現有上方 tab navigation 中新增一個 tab：

```text
使用者建議與改善
```

建議 tab 順序：

```text
EE/RF/PWR-專案元件清單
TH/ME-專案元件清單
模擬與實測驗證
凸台&TIM尺寸速算
使用者建議與改善
```

此 tab 被選取時需顯示藍色 active 狀態，風格請比照現有 tab。

---

## 3. 頁面標題區

進入此分頁後，頁面最上方顯示：

```text
使用者建議與改善
```

副標題：

```text
記錄跨部門使用者在操作工具時遇到的問題、改善建議與功能需求
```

建議左側可放一個 feedback / speech bubble 類型 icon。

---

## 4. 頁面整體結構

此頁面分成 4 個主要區塊：

```text
使用者建議與改善
│
├─ 1. 快速回饋分類
│
├─ 2. 新增回饋
│
├─ 3. 回饋統計摘要
│
└─ 4. 回饋清單與追蹤狀態
```

建議以現有工具的白底卡片、圓角、淡陰影、藍色主色系設計。

---

## 5. 區塊 1：快速回饋分類

### 5.1 功能目的

讓使用者可以快速選擇回饋類型，點擊後自動帶入「新增回饋」表單中的 `回饋類型` 欄位。

### 5.2 UI 配置

一列 5 張卡片式按鈕：

| 顯示文字 | 對應回饋類型 |
|---|---|
| 回報 Bug | Bug |
| 提出改善建議 | 改善建議 |
| 新功能需求 | 新功能需求 |
| 資料欄位問題 | 資料欄位問題 |
| 權限/登入/儲存問題 | 權限/登入/儲存問題 |

每張卡片建議包含：

- icon
- 文字
- 右側 chevron 箭頭
- hover 效果

### 5.3 行為

點擊卡片時：

1. 自動設定 `feedback_type`
2. 將焦點移到「新增回饋」表單
3. 不需立即送出資料

---

## 6. 區塊 2：新增回饋

### 6.1 功能目的

讓使用者新增一筆回饋紀錄。

### 6.2 表單欄位

請依照最後 UI 概念圖，建立以下欄位。

| 欄位名稱 | Key | 類型 | 必填 | 說明 |
|---|---|---|---|---|
| 回饋類型 | type | select | Y | Bug / 改善建議 / 新功能需求 / 資料欄位問題 / 權限/登入/儲存問題 / 其他 |
| 發生分頁 | tab | select | Y | 問題或建議發生在哪個工具分頁 |
| 標題 | title | text | Y | 一句話描述，建議 50 字以內 |
| 優先度 | priority | select | Y | 低 / 中 / 高 / 緊急 |
| 提出人 | reporter | text | Y | 可預設目前登入者名稱，或手動輸入 |
| 聯絡方式 | contact | text | N | Email 或 Teams ID |
| 詳細說明 | description | textarea | Y | 描述問題、建議、情境、重現步驟或錯誤訊息 |
| 目前狀況 | current_behavior | textarea | N | 現在工具的行為或問題 |
| 期望結果 | expected_behavior | textarea | N | 希望工具怎麼改善 |
| 截圖或附件連結 | attachment_link | text | N | 可貼 SharePoint / Teams / 圖片連結，多筆用逗號分隔 |
| 備註 | note | textarea | N | 其他補充 |

### 6.3 發生分頁選項

```text
EE/RF/PWR-專案元件清單
TH/ME-專案元件清單
模擬與實測驗證
凸台&TIM尺寸速算
使用者建議與改善
SharePoint登入/儲存
整體工具
其他
```

### 6.4 優先度選項

```text
低
中
高
緊急
```

### 6.5 表單按鈕

右下角放兩個按鈕：

```text
清除
送出回饋
```

`送出回饋` 為主要按鈕，使用藍色。  
`清除` 為次要按鈕，使用灰色或白底外框。

### 6.6 送出行為

按下 `送出回饋` 時：

1. 檢查必填欄位：
   - 回饋類型
   - 發生分頁
   - 標題
   - 優先度
   - 提出人
   - 詳細說明

2. 若缺少必填欄位，顯示提示，不得送出。

3. 若通過驗證，建立一筆 feedback item。

4. 自動帶入：
   - `created_at`
   - `updated_at`
   - `status = "待處理"`

5. 儲存到資料庫。

6. 儲存成功後：
   - 顯示成功訊息
   - 清空表單
   - 重新整理回饋清單
   - 更新統計摘要

### 7.1 功能目的

```text
新增    12

修正中   3
已完成  18
```

### 7.4 顏色建議

| 狀態 | 顏色 |

| 評估中 | Orange |
| 修正中 | Green |
| 已完成 | Purple |

## 8. 區塊 4：回饋清單與追蹤狀態

### 8.1 功能目的

顯示所有使用者提交的回饋，並讓維護者追蹤狀態。

### 8.2 注意：不要顯示 ID 與負責人欄位

根據最後 UI 調整，此表格不要顯示：

```text
ID
負責人
```

即使資料內部有 id，也不需要顯示在表格上。

### 8.3 表格欄位

請顯示以下欄位：

| 欄位名稱 | Key | 說明 |
|---|---|---|
| 日期 | created_at | 建立時間，格式 `YYYY/MM/DD HH:mm` |
| 類型 | type | Bug / 改善建議 / 新功能需求等 |
| 分頁 | tab | 發生分頁 |
| 標題 | title | 回饋摘要 |
| 提出人 | reporter | 提出者 |
| 優先度 | priority | 低 / 中 / 高 / 緊急 |
| 狀態 | status | 待處理 / 評估中 / 修正中 / 已完成 / 不採用 |
| 完成日期 | closed_at | 未完成顯示 `—` |
| 處理備註 | admin_note | 維護者處理備註 |

表格欄位順序：

```text
日期 | 類型 | 分頁 | 標題 | 提出人 | 優先度 | 狀態 | 完成日期 | 處理備註
```

### 8.4 表格範例資料

請用以下類型資料作為測試或初始示意：

```text
2024/05/22 10:15 | Bug | 模擬與實測驗證 | 上傳 S2P 檔案後分析結果未更新 | 張工程師（RF部） | 高 | 待處理 | — | 已回報問題，待評估
```

## 10. 狀態設計

### 10.1 狀態選項

請使用以下狀態：

```text
待處理
評估中
已排入
修正中
已完成
不採用
```

### 10.2 狀態意義

| 狀態 | 意義 |
|---|---|
| 待處理 | 使用者剛提交，尚未處理 |
| 評估中 | 維護者正在評估是否要處理 |
| 已排入 | 確認會處理，但尚未開始 |
| 修正中 | 已開始修改 |
| 已完成 | 已修正並上線或已完成改善 |
| 不採用 | 評估後不執行，需填寫原因 |

### 10.3 狀態顏色

| 狀態 | Badge 顏色建議 |
|---|---|
| 待處理 | Blue |
| 評估中 | Orange |
| 已排入 | Indigo / Blue-gray |
| 修正中 | Green |
| 已完成 | Purple |
| 不採用 | Gray |

- 修改完成後需儲存到 SharePoint JSON DB

## 13. 資料儲存設計

### 13.1 新增 collection

請在現有 JSON DB 新增 collection：

```json
"feedback_items": {}
```

### 13.2 資料格式

每一筆 feedback item 建議格式：

```json
{
  "type": "Bug",
  "tab": "模擬與實測驗證",
  "title": "上傳 S2P 檔案後分析結果未更新",
  "priority": "高",
  "reporter": "張工程師（RF部）",
  "contact": "chang.engineer@company.com",
  "description": "上傳 S2P 檔案後，分析結果沒有重新計算，需要重新整理頁面才會更新。",
  "current_behavior": "檔案已上傳，但結果仍顯示舊資料。",
  "expected_behavior": "上傳後應自動重新分析並更新結果。",
  "attachment_link": "",
  "note": "",
  "status": "待處理",
  "closed_at": "",
  "admin_note": "已回報問題，待評估",
  "created_at": "2026-05-07T10:30:00+08:00",
  "updated_at": "2026-05-07T10:30:00+08:00"
}
```

### 13.3 內部 ID 產生方式

雖然表格不顯示 ID，但內部仍需要 key。

建議 key 格式：

```text
FB-YYYYMMDD-HHmmss-random
```

例如：

```text
FB-20260507-103015-A3F2
```

此 ID 僅作為資料庫 key，不顯示在 UI 表格。

## 14. SharePoint / dbAdapter 整合

請沿用現有 `dbAdapter` 的介面。

新增資料時：

1. 讀取現有 `feedback_items`
2. 新增一筆 item
3. 使用 `dbAdapter.updateDoc()` 或等效方式更新 JSON DB
4. 重新 render 清單

如果目前 DB 是單一 JSON 結構，建議確保：

```json
{
  "feedback_items": {
    "FB-20260507-103015-A3F2": {
      ...
    }
  }
}
```

若 `feedback_items` 不存在，請自動建立空物件。

## 20. 開發注意事項

1. 不要把此功能做成 GitHub issue 串接，第一版只需要寫入目前 SharePoint JSON DB。
2. 不要顯示內部 ID。
3. 不要顯示負責人欄位。
4. 表格中可保留 `處理備註`，讓維護者記錄處理狀態。
5. 若目前工具有密碼解鎖 / lock 機制，請沿用。
6. 優先確保資料可儲存與 reload 後可恢復。
7. UI 先求穩定清楚，不需要做複雜動畫。
8. 新增的程式碼請避免破壞現有 Tab 1、Tab 2、Tab 3、Calculator 功能。

## 21. 最終目的

此分頁要讓跨部門使用者在使用工具時，可以即時記錄問題與改善建議，避免回饋散落在 Teams、Email 或口頭討論中。

工具維護者則可以透過此頁面追蹤：

- 有哪些 Bug
- 哪些建議已完成
- 哪些需求仍在評估
- 哪些功能需要排入後續版本

第一版重點是：

```text
可以新增、可以保存、可以查看、可以篩選、可以追蹤狀態。
```
